# delivery-closeout

## 用途

把内部执行结果收成用户可直接消费的交付格式，避免用户从过程噪音里自己找结论。

## 何时读取

当任务即将结束，且已经有产出或阶段结论时读取。

常见适用场景：

- 工程任务做完一轮后
- 调研已拿到可交付结论
- 长任务阶段性收口
- 多智能体结果已经回流，需要主 agent 对外汇总

## 默认输出结构

- 现在状态
- 结果
- 已验证 / 未验证
- 是否需要用户决策

## 简版模板

```md
- 现在状态：
- 结果：
- 已验证 / 未验证：
- 是否需要用户决策：
```

## 使用原则

- 先给结论，再补必要依据
- 用户只需要知道的过程才保留
- 已验证和未验证必须分开写
- 需要用户拍板时，明确指出决策点
- 不把内部执行备注、失败日志、工具噪音原样外发

## 输出建议

### 适合短回复的情况
直接按 4 项简版输出。

### 适合汇报型回复的情况
先一句话结论，再用 4 项结构收口。

## 常见误区

### 误区 1：过程写太多，结论埋在中间
错因：用户还得自己做二次提炼。

改法：先给结果，再给必要说明。

### 误区 2：已做和已验证混在一起
错因：用户会误以为已经完全落地。

改法：把“已验证 / 未验证”单列。

### 误区 3：明明需要用户决定，却写成通知式汇报
错因：任务看起来结束了，实际上停在决策口。

改法：明确写“是否需要用户决策”以及要决什么。

## 结果标准

用户看完回复后，不需要重新翻过程，就能知道现在到了哪一步、结果是什么、是否真的验证过、自己要不要做决定。
