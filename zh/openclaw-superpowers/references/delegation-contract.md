# delegation-contract

## 用途

凡是需要多智能体分派时，默认补齐“四件套”，把口头交代升级成可执行的委派契约。

## 何时读取

当主 agent 需要调用以下任一类型的协作角色，且任务不属于一次性轻量补充时读取：

- `implementer`
- `reviewer`
- `writer`
- `researcher`

如果宿主环境使用的是别的角色名，也按同样原则套用。

以下情况可弱化，不必机械补满：

- 单次很小的补充查询
- 明确只有一个极窄动作的子任务
- 主 agent 已经在上下文里把边界说得非常死

## 四件套

每次委派默认写清：

- 目标
- 完成标准
- 验证要求
- 回流格式

## 标准模板

```md
目标：
完成标准：
验证要求：
回流格式：
```

## 各项要求

### 目标
说清楚这次子智能体到底负责什么，不要让它自己猜主线。

### 完成标准
写清交付到什么程度才算完成，避免“做了一点也算交差”。

### 验证要求
说明需要什么证据：代码、运行结果、截图、日志、文档结构检查、交叉核对等。

### 回流格式
约定回来的内容格式，至少包含：做了什么、证据是什么、还有什么没完成。

## 推荐回流格式

```md
- 已完成：
- 验证证据：
- 未完成 / 风险：
- 建议下一步：
```

## 使用原则

- 简单补充型子任务可以弱化，但复杂任务不要省
- 不要求每次字面写满四行，关键是四件信息不能缺位
- implementer 负责实现，不给自己背最终质量书
- reviewer 负责验证，不代替 implementer 做主体实现
- writer / researcher 也一样需要明确产出物和回流格式
## 常见误区

### 误区 1：只说“你去做一下”
错因：子智能体只能靠猜，回来的结果大概率偏题。

改法：至少补齐目标 + 完成标准。

### 误区 2：让 reviewer 既实现又验收
错因：写审分离失效，假进展更难发现。

改法：实现和验收分开。

### 误区 3：子任务做完了，但主 agent 还要重新追问细节
错因：回流格式没提前约定。

改法：先写回流格式，再发任务。

## 结果标准

委派完成后，子智能体应能直接开工，不需要再追问“做什么、怎么证明、怎么回报”。
