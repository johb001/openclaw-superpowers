# checkpoint-recap

## 用途

长任务推进过程中，按阶段收口关键状态，避免失焦、断层和续做困难。

## 何时读取

当任务满足任一条件时读取：

- 耗时较长
- 明显多阶段
- 已开团队
- 中途卡住
- 用户切话题后再回来
- 上下文已经明显拉长

## 默认输出结构

- 当前目标
- 已完成
- 当前卡点
- 下一步

## 简版模板

```md
- 当前目标：
- 已完成：
- 当前卡点：
- 下一步：
```

## 使用原则

- 只收关键状态，不复述全部过程
- 不是每轮都写 checkpoint，只有阶段变化明显时才收口
- 写给“下一个接手的人”和“未来的自己”都能看懂
- 卡点要写成阻碍，不写成情绪
- 下一步要可执行，不能只写“继续推进”

## 什么时候必须收口

以下情况建议立即做一次 checkpoint：

1. 刚完成一个阶段，准备进入下一段
2. 遇到阻碍，短时间内过不去
3. 任务要跨回复继续
4. 调用了多个子智能体，需要主 agent 收束现场
5. 用户中途插入新话题，原任务可能稍后再接

## 常见误区

### 误区 1：把 checkpoint 写成流水账
错因：信息很多，但没法快速恢复现场。

改法：只保留“目标 / 已完成 / 卡点 / 下一步”。

### 误区 2：只写做了什么，不写为什么停在这里
错因：后续续做时又得重新排查。

改法：卡点要明确写出当前阻碍。

### 误区 3：下一步写得太空
错因：看起来在推进，实际上没有行动入口。

改法：下一步写成可执行动作。

## 结果标准

任何阶段中断后，都能靠这四项快速恢复上下文并继续推进，而不是重新翻整段历史消息。
