# task-shaping

## 用途

在复杂任务开始前，把任务压缩成清晰、可执行、可验收的定义，避免没对齐就直接开干。

## 何时读取

当满足任一条件时读取：

- 用户目标模糊
- 任务明显多步骤
- 涉及实现 / 配置 / 自动化 / 项目推进，且范围还没压清
- 有高返工风险
- 用户明确说“先规划一下”“先理一理”“先别做，先想清楚”

以下情况通常不需要单独读取：

- 用户目标已经非常清楚
- 任务一步就能完成
- 只是小修小改，且完成标准天然明确

## 输出结构

默认按以下结构整理：

- 目标
- 完成标准
- 边界
- 风险 / 不确定点
- 成功判定

## 最低要求

在进入执行前，至少能回答清楚：

1. 到底要做什么
2. 做到什么算完成
3. 哪些内容不在本次范围内
4. 哪些点需要先确认或带着风险推进

## 简版模板

```md
- 目标：
- 完成标准：
- 边界：
- 风险 / 不确定点：
- 成功判定：
```

## 使用原则

- 能一句话说清的小任务，不强行套模板
- 目标已经很清楚时，只补缺的那一项
- 不必每次 5 项全写；缺什么补什么
- 复杂度越高，越要先把“完成标准”写死
- 如果边界不清，不要直接开始重活
## 常见误区

### 误区 1：只复述用户原话，不做压缩
错因：看起来在对齐，实际上还是模糊。

改法：把用户原话压成可执行定义，而不是原样重复。

### 误区 2：只写要做什么，不写做到什么算完
错因：容易中途失焦，最后也没法验收。

改法：明确“完成标准”和“成功判定”。

### 误区 3：边做边想，最后越做越大
错因：范围不断膨胀，任务容易拖住。

改法：先写边界，明确哪些不做。

## 结果标准

当 task-shaping 做完后，主 agent 应该能自然进入下一步，而不是继续围着“到底要不要做这个”打转。
