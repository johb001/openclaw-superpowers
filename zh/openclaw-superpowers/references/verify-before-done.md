# verify-before-done

## 用途

在对外宣布完成前，强制做最小验证，防止把“改了”误报成“成了”。

## 何时读取

当任务涉及以下任一类型时读取：

- 代码实现
- 配置修改
- bug 修复
- 文档高风险编辑
- 自动化落地
- 系统操作
- 外部状态变更

## 默认输出结构

- 改了什么
- 验证方式
- 验证结果
- 未验证项
- 是否可按完成汇报

## 最低验证要求

至少完成一轮最小真实验证。根据场景可选：

- build / lint / test
- 实际运行结果
- 页面可见
- 配置真实调用通过
- 文档结构重新读取确认
- 日志 / 返回值 / 状态检查

如果受环境限制无法完整验证，也必须明确：

- 已验证到哪一层
- 哪一层还没验证
- 为什么暂时不能继续验证

## 拦截规则

如果没有验证，不要直接对外说：

- 已完成
- 已修复
- 可以了
- 没问题
- 已落地

此时应该明确说：

- 已修改，但未验证
- 已做最小处理，待验证
- 目前只确认到代码 / 配置层，未确认真实生效

## 简版模板

```md
- 改了什么：
- 验证方式：
- 验证结果：
- 未验证项：
- 是否可按完成汇报：
```

## 使用原则

- 验证优先真实链路，不只看静态代码
- 没法完整验证时，也要明确“验证到哪一层”
- 对外汇报要区分：已验证 / 高概率 / 未验证
- 不能把“理论上可行”写成“已经完成”
- 没证据就降级表达，不硬报喜
## 常见误区

### 误区 1：代码写完就算完成
错因：代码落地不等于功能生效。

改法：至少补一轮运行级验证。

### 误区 2：凭感觉判断“应该可以”
错因：没有证据，风险会转嫁给用户。

改法：只按已验证事实说话。

### 误区 3：高风险文档或配置操作做完不回看
错因：看起来动了，实际可能改坏了。

改法：执行后重新读取 / 检查现状。

## 结果标准

对外给出“已完成”前，必须至少有一条最小证据链支撑这个结论。
